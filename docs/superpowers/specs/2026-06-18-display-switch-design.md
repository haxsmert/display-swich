# 显示器开关小工具 —— 设计文档

- 日期:2026-06-18
- 状态:已通过可行性验证,待实现
- 目标平台:Apple Silicon(M 系列)+ macOS 26（开发机:M5 Pro / macOS 26.5.1）

## 1. 背景与目标

Mac 外接双显示器场景下,希望用一个**极简**的菜单栏小工具,**软件层面关闭或打开其中一块外接显示器**——做到 BetterDisplay 的「断开/重连」核心效果,但不要它的复杂度和付费。

「关闭」的明确定义:**虚拟断开**——让 macOS 以为该显示器被拔掉(从活跃显示器列表消失、窗口自动迁移到另一块屏、菜单栏/Dock 重排),「打开」时再让它干净地回来。不是熄背光、不是镜像切换。

### 两种使用场景（工具需自适应）

- **MacBook 外接**:有时希望把**外接屏全部关黑**(只想让外接屏全暗下来)。此时内建屏是兜底——开盖即可用内建屏操作恢复(未来也可配快捷键)。所以这种机器允许外接屏全关。
- **Mac mini / Studio 等无内建屏**:没有兜底屏,必须始终保留至少一块活跃外接屏,否则会陷入完全无显示、无法操作的死局。

保护规则因此是「**机型感知**」的(见第 7 节),而非简单「至少留一块」。

## 2. 可行性验证结论（已完成）

在开发机(M5 Pro / macOS 26.5.1 / 双小米 4K)上做过两步验证:

- **只读探测:** 能可靠枚举两块屏并分清主/副屏;关键私有符号 `CGSConfigureDisplayEnabled`、`SLSConfigureDisplayEnabled` 在 macOS 26 上均存在;公开恢复兜底 `CGRestorePermanentDisplayConfiguration` 也在。
- **断开+恢复实测:** 对**非主屏**执行断开 → 活跃屏 2→1(真的从系统消失);执行恢复 → 活跃屏 1→2(干净回来)。两个方向调用都返回成功。

**结论:纯自研「真·断开」在该平台可行**,不需要 BetterDisplay、不需要关 SIP、不需要特殊 entitlement(开发者签名即可运行)。

⚠️ 平台限制:此结论依赖 Apple Silicon + 较新 macOS。Intel 平台不在目标范围(该路径在 Intel 上不稳定、可能不可逆)。

## 4. 架构总览

一个 macOS 菜单栏常驻应用(`LSUIElement = true`,无 Dock 图标、无主窗口),Swift + AppKit,**零第三方依赖**。

交互:点菜单栏图标 → 弹出菜单,动态列出当前所有**外接**显示器,每块一行带勾选(✓ = 当前开启);点某行即切换该屏开/关;菜单底部「退出」。

### 模块拆分（各自职责单一、可独立理解与测试）

| 模块 | 职责 | 依赖 |
|---|---|---|
| `DisplayInfo` | 值类型:`displayID` / `uuid` / 名称 / `bounds` / 是否主屏 / 是否内建 / 是否启用 | 无 |
| `DisplayController` | 枚举外接显示器、计算左右命名、执行开/关、保护规则判定 | CoreGraphics(私有符号薄封装) |
| `StatusMenuController` | 管理 `NSStatusItem`,把 `DisplayController` 的状态渲染成菜单项,处理点击与刷新 | AppKit、`DisplayController` |
| `AppDelegate` | 启动装配、退出时恢复兜底、显示器热插拔时刷新菜单 | AppKit |

边界说明:`DisplayController` 不碰 UI;`StatusMenuController` 只读 `DisplayController` 暴露的状态与方法,不直接调系统显示 API。这样开关逻辑可在不起 UI 的情况下单测。

## 5. 显示器开/关机制

私有符号通过 `dlsym(RTLD_DEFAULT, ...)` 解析后 `unsafeBitCast` 成 `@convention(c)` 函数指针调用,避免链接期硬依赖。

开/关的核心调用(包在公开的配置事务里):

```
CGBeginDisplayConfiguration(&cfg)
CGSConfigureDisplayEnabled(cfg, displayID, on)   // 私有符号
CGCompleteDisplayConfiguration(cfg, <option>)
```

### 退出恢复策略（关键安全决策）

目标:**app 退出或崩溃时,所有被关的屏自动恢复**,永不出现「屏黑着却恢复不了」的死局。

- **首选 `forAppOnly`**:配置仅在本进程存活期间有效,进程退出(含崩溃)由系统自动回滚 → 天然满足目标。
- **实现时当场验证**:`forAppOnly` 下「断开」是否仍**全局生效**(让窗口迁移、活跃屏减少)存在不确定性。写代码时先实测一次。
- **兜底方案(若 `forAppOnly` 不全局生效)**:改用已验证可行的 `forSession`,并辅以:
  1. `applicationWillTerminate` 中主动 re-enable 所有被本 app 关掉的屏 +  `CGRestorePermanentDisplayConfiguration()`;
  2. app **启动时**先把所有屏恢复一遍作为兜底(防上次崩溃残留);
  3. `forSession` 本身在注销/重启时也会恢复。

不论走哪条,都**不使用 `permanently`**(那是「断开后无法用代码恢复」死坑的来源)。

## 6. 显示器标识与命名

- **稳定标识用 UUID**:两块小米屏的 vendor/model/serial 完全相同,无法据此区分;`displayID` 会随重连变化。`CGDisplayCreateUUIDFromDisplayID` 得到的 UUID 在重连后稳定,作为持久标识。
- **菜单显示名 = 系统名 + 左右 + 主屏标记**:读 `CGDisplayBounds(id).origin.x` 判定左右,主屏额外标注。示例:`Mi Monitor（左·主屏）` / `Mi Monitor（右）`。系统名取不到时回退为 `显示器 #<序号>`。

## 7. 安全保护规则

核心目标:保证系统永远至少有一条**可用于操作恢复的显示路径**,绝不陷入完全无显示、无法操作的死局。`DisplayController` 在执行「关闭」前强制校验,违反则拒绝并(若有 UI)给出提示:

1. **内建屏永不关闭、不纳入开关**:仅列出与操作外接屏。
2. **机型感知的「关闭外接屏」保护**:
   - 若这台 Mac **有内建屏**(MacBook 等便携机):允许把外接屏**全部关掉**——开盖即可用内建屏操作恢复(配合未来的快捷键)。内建屏构成兜底恢复路径,即使当前合盖、内建屏未激活也算数(开盖即可用)。
   - 若这台 Mac **无内建屏**(Mac mini / Studio / Mac Pro 等):必须**至少保留一块活跃外接屏**,禁止关掉最后一块活跃外接屏。
3. 关闭目标必须是当前在线且启用的外接屏。

**「是否有内建屏」的检测**:基于「机器是否**具备**内建屏」而非「内建屏当前是否活跃」(合盖时内建屏不在活跃列表,但仍是有效兜底)。实现时确定可靠手段(候选:IOKit 检测内建显示面板存在性 / 是否为带电池的便携机)。**检测不确定时,保守按「无内建屏」处理**——宁可禁止全关,也不冒全黑风险。

## 8. 构建与分发

- **Swift Package Manager** 可执行目标,`swift build` / `swift run` 可构建运行。
- 一个**打包脚本**(`make app` 或 `scripts/package.sh`)把产物包装成 `.app`:生成 `Info.plist`(含 `LSUIElement = true`)、放入 bundle、本地开发者签名。
- 仅本地/自分发,不上架 App Store(用到私有 API,本就无法上架——与目标一致)。

## 9. 测试策略

- **单元测试(自动化)**:针对 `DisplayController` 的纯逻辑——保护规则(剩 1 块时拒关)、左右命名计算、外接屏筛选、目标合法性——通过注入假的 `DisplayInfo` 列表测试,不触碰真实硬件。
- **Smoke 测试(手动)**:真实开/关屏因有硬件副作用,沿用已验证的脚本流程手动验证一次(断开→活跃屏减少→恢复→活跃屏复原)。
- 系统显示 API 调用集中在 `DisplayController` 一处薄封装,便于隔离。

## 10. 已知风险与缓解

| 风险 | 缓解 |
|---|---|
| 私有符号跨 macOS 大版本可能改名/失效 | 启动时探测符号是否存在,缺失则禁用开关并提示;集中在一处便于适配 |
| `forAppOnly` 可能不让断开全局生效 | 实现时当场实测,有 `forSession` + 多重恢复兜底 |
| app 崩溃残留黑屏 | 首选 `forAppOnly`(系统自动回滚);兜底走启动时恢复 + 注销恢复 |
| 无内建屏机型上关到 0 块活跃屏 | 机型感知保护:无内建屏时至少保留一块活跃外接屏 |
| 「是否有内建屏」检测不准 | 实现时确认可靠手段;不确定时保守按「无内建屏」处理(更安全,宁可禁止全关) |
| 显示器热插拔导致菜单状态过期 | 监听 `CGDisplayReconfiguration` 回调刷新菜单 |
