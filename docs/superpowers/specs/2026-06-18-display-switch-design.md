# 显示器开关小工具 —— 设计文档

- 日期:2026-06-18(2026-06-22 修订:统一模型 + 死锁安全模型定稿 + 机型感知 + 架构自检)
- 状态:**v1.0 已完成、真机走查通过**(列表/系统名/关一块/最后一块灰显/关内建留外接/恢复/编号稳定 全过);机型感知保护 + 死锁加固 + `isSupported`(私有符号 + Apple Silicon 架构)自检 + app 图标 均已落地
- 目标平台:Apple Silicon(M 系列)+ macOS 13+(开发/验证机:M5 Pro / macOS 26.5.1);非 Apple Silicon 由 `isSupported` 判不支持(只读)

## 1. 背景与目标

Mac 外接双显示器场景下,希望用一个**极简**的菜单栏小工具,**软件层面关闭或打开某块显示器**——做到 BetterDisplay 的「断开/重连」核心效果,但不要它的复杂度和付费。

「关闭」的明确定义:**虚拟断开**——让 macOS 以为该显示器被拔掉(从活跃显示器列表消失、窗口自动迁移到另一块屏、菜单栏/Dock 重排),「打开」时再让它干净地回来。不是熄背光、不是镜像切换。

### 统一模型(一句话讲清)

菜单列出**所有活跃屏(含内建屏)**,显示名 = 系统名 + UUID 稳定编号 +(主);任意屏都可关,但**绝不把用户置于「无法恢复的全黑」**——保护规则见 §7(机型感知:有内建屏的笔记本允许关到全黑、靠开盖恢复;无内建屏的台式机必须至少留一块活跃屏)。

> 演进说明:早期按「左/右」命名;2026-06-22 重构为统一模型(内建屏纳入菜单、UUID 稳定编号),中途一度简化保护为「至少留一块活跃屏」,后又按真实使用场景**升级回机型感知**(笔记本合盖用外接、关光外接靠开盖恢复内建)并加死锁加固。本文档已按最新状态描述。

## 2. 可行性验证结论（已完成 / 已真机复验）

在开发机(M5 Pro / macOS 26.5.1 / 双小米 4K + 内建屏)上验证:

- **只读探测:** 能可靠枚举所有屏并分清主/副/内建;关键私有符号 `CGSConfigureDisplayEnabled` 在 macOS 26 上存在;公开恢复兜底 `CGRestorePermanentDisplayConfiguration` 也在。`CGDisplayCreateUUIDFromDisplayID` 需 `import ColorSync`。
- **断开+恢复实测:** 对非主屏、对内建屏(即便它当前是主屏)执行 `.forAppOnly` 断开 → 活跃屏减少(真的从系统消失);恢复 → 干净回来。`.forAppOnly` 的断开**全局生效**(窗口迁移、活跃屏减少),已实测确认。

**结论:纯自研「真·断开」在该平台可行**,不需要 BetterDisplay、不需要关 SIP、不需要特殊 entitlement(开发者签名即可运行)。

⚠️ 平台限制:此结论依赖 Apple Silicon + 较新 macOS。Intel 平台不在目标范围。**已加架构自检**:`isSupported` 要求 `hw.optional.arm64 == 1`,非 Apple Silicon 一律判不支持(只读不动屏);加之 app 实为 arm64-only、Intel 根本无法启动,双重保证不在未验证平台上动显示器。

## 3. 非目标（YAGNI,本期不做）

- **开机自启动 —— 永不做(这是安全约束,见 §7,不是临时取舍)**
- 记忆/持久化上次的开关状态
- 全局快捷键 / 急救快捷键
- 自动触发(如插电源/接屏时自动切)
- 拔屏自动恢复守护(已实验证伪,见 §7)
- 显示器重命名
- 熄背光/亮度/DDC 控制 / 创建虚拟显示器 / Intel 平台支持

以上都不实现。需要时再开新一轮设计。

## 4. 架构总览

一个 macOS 菜单栏常驻应用(`LSUIElement = true`,无 Dock 图标、无主窗口),Swift + AppKit,**零第三方依赖**。

交互:点菜单栏图标 → 弹出菜单,动态列出当前**所有**显示器(含内建),每块一行带勾选(✓ = 当前开启);点某行即切换该屏开/关;只剩最后一块活跃屏时该行灰显不可关;菜单底部「退出」。

| 模块 | 职责 | 依赖 |
|---|---|---|
| `DisplayInfo` | 值类型:`id` / `uuid` / 名称 / `bounds` / `isMain` / `isBuiltin` / `isActive` | 无 |
| `DisplayController` | 枚举所有屏、系统名+(主)命名、执行开/关、「至少留一块活跃屏」保护判定、维护已关闭屏快照 | CoreGraphics(私有符号薄封装) |
| `StatusMenuController` | 管理 `NSStatusItem`,把 `DisplayController` 状态渲染成菜单项,处理点击与刷新 | AppKit、`DisplayController` |
| `AppDelegate` | 启动装配(启动即恢复一遍作兜底)、退出时恢复全部、热插拔时刷新菜单 | AppKit |

边界:`DisplayController` 不碰 UI;`StatusMenuController` 只读 `DisplayController` 暴露的状态与方法,不直接调系统显示 API。开关逻辑可在不起 UI 下单测。

## 5. 显示器开/关机制

私有符号通过 `dlsym(RTLD_DEFAULT, ...)` 解析后 `unsafeBitCast` 成 `@convention(c)` 函数指针调用,避免链接期硬依赖。

```
CGBeginDisplayConfiguration(&cfg)
CGSConfigureDisplayEnabled(cfg, displayID, on)   // 私有符号
// 失败:CGCancelDisplayConfiguration(cfg) 回滚,不泄漏事务
CGCompleteDisplayConfiguration(cfg, .forAppOnly)
```

### 退出恢复策略（关键安全决策）

目标:**app 退出或崩溃时,所有被关的屏自动恢复**,永不出现「屏黑着却恢复不了」的死局。

- **固定用 `.forAppOnly`**:配置仅在本进程存活期间有效,进程退出(含崩溃、被杀、强制重启)由系统自动回滚。已实测:`.forAppOnly` 下断开仍全局生效。
- **多重兜底**:`applicationWillTerminate` 主动 re-enable 所有被本 app 关掉的屏 + `CGRestorePermanentDisplayConfiguration()`;app **启动时**先恢复一遍(防上次残留)。
- **绝不使用 `.permanently`**(那是「断开后无法用代码恢复」死坑的来源)。

## 6. 显示器标识与命名

- **稳定标识用 UUID**:两块同型号屏的 vendor/model/serial 完全相同,无法据此区分;`displayID` 会随重连变化(实测见过 2↔5 互换)。`CGDisplayCreateUUIDFromDisplayID` 得到的 UUID 在重连后稳定,作持久标识。
- **菜单显示名 = 基名 + 自编号 +(主)**:系统名末尾的 `(N)` 是「**动态消歧**」——仅当同名屏同时在场才出现,关掉其一就消失,**不稳定**(实测:关掉一块,剩下那块的 `(2)` 会掉成光秃秃的名字;重开后还可能因 macOS 重排布而 `(2)↔(1)` 漂移)。故剥掉系统这个后缀取**基名**,再对同名多块**按 UUID 稳定编号** `(1)(2)…`——**编号绑定唯一标识、与屏的位置无关**,关/开/主屏转移都不漂移;同名屏唯一时不编号。主屏额外标 `(主)`,基名取不到时回退 `显示器`。菜单按「基名 + UUID」排序,同名屏相邻且编号顺序排列。已关闭的屏参与编号(整组含已关闭快照,故关掉其一不丢号、不被顶替),但重建菜单时清掉 `isMain`(已关的屏不可能是主屏,避免「两块都标主」)。

## 7. 安全保护规则与死锁安全模型

### 7.1 软件保护:机型感知,绝不陷入「无法恢复的全黑」

`DisplayController` 在执行「关闭」前强制校验 `canDisable(target, among: active, builtInFallback:)`,违反则拒绝(UI 上该行灰显):

- 关掉后仍剩 **≥1 块活跃屏** → 允许;
- 关掉后会变成 **0 块活跃(全黑)** → 仅当存在「**可开盖恢复的内建屏**」兜底才允许:
  - **有内建屏的机器**(笔记本,合盖、内建屏不活跃也算)→ **允许关到全黑**——开盖即可硬件恢复内建屏。满足「合盖用外接、关光外接、开盖恢复」的真实场景。
  - **无内建屏的机器**(Mac mini / Studio)→ **禁止**关最后一块活跃屏(无恢复路径)。

**死锁加固**:`builtInFallback` 只在内建屏面板存在**且未被本 app 软件关闭**时为真——被软件关掉的内建屏开盖也救不回,不算兜底。此外若 `target` 自身就是内建屏,关掉它后内建屏即被软件关闭,故也禁止把内建屏自己关成全黑。两者合起来堵死「软件关内建 + 关光外接 → 开盖救不回」的死锁。

「是否有内建屏面板」用 **`hasBuiltInDisplay()`** 判定:先看在线列表有无内建屏,再用「是否带内建电池」作便携机代理(合盖时内建屏不在在线列表);**检测不确定时保守按「无内建屏」处理**(宁可禁止全关,不冒险)。

### 7.2 死锁边界与逃生(2026-06-22 定稿)

§7.1 的加固已**主动堵死「纯软件」走到全黑**(软件关内建后不许再关光外接;也不许把内建自己关成全黑)。但软件**挡不住物理拔屏**,仍存在一个边界:

> **软件关掉内建屏 + 保留外接屏活跃,随后物理拔掉所有外接屏 → 0 块活跃屏 → 持续全黑。**

**已实验证伪「同进程内软件自动恢复」这条路:** 全黑发生时,显示子系统进入冻结态——`CGGetActiveDisplayList` 返回陈旧缓存(轮询失灵),`CGDisplayRegisterReconfigurationCallback` 的拔屏事件也不派发(回调失灵)。两条探测路径在最需要的那一刻同时哑火。因此「检测到拔屏再补救」的守护机制**不可靠,本期不做**(连同急救快捷键、IOKit 底层探测一并不做)。

**采用的逃生方案 = 强制重启,成立依赖两条硬不变量:**

1. **全程 `.forAppOnly`、绝不 `.permanently`**:进程一死(强制重启会杀掉它),系统自动回滚所有被关的屏 → 重启后显示配置必为正常。
2. **永不开机自启**:重启后 app 不会自己拉起,也就不会重新关屏 → 重启后必为「所有屏正常、app 未运行」的干净状态。

两条缺一不可。「永不开机自启」因此是**死锁逃生链条的一环、硬安全约束**,不是可选偏好——任何引入开机自启的改动都会破坏这条逃生路径,禁止。

> 取舍说明:接受「死锁时短暂全黑、需强制重启」这一代价,换取实现的极简与可靠;不为一个边缘场景引入证伪过的不可靠守护。

## 8. 构建与分发

- **Swift Package Manager** 可执行目标,`swift build` / `swift run` 可构建运行。
- 打包脚本(`scripts/package.sh`)把产物包成 `.app`:生成 `Info.plist`(含 `LSUIElement = true`)、放入 bundle、本地开发者签名(ad-hoc)。
- 仅本地/自分发,不上架(用到私有 API,本就无法上架——与目标一致)。

## 9. 测试策略

- **单元测试(自动化)**:针对 `DisplayController` 纯逻辑——保护规则(机型感知:无内建屏剩 1 块拒关、有内建屏允许关到全黑、软件关内建后防死锁)、UUID 稳定编号、已关闭屏清主屏标记、`isSupported` 兜底、目标合法性——注入假 `DisplayInfo` 列表测试,不碰真实硬件。
- **Smoke / 真机走查(手动,需用户在场)**:真实开/关屏有硬件副作用,用脚本或打包后的 app 手动验证(列表/命名/关一块/最后一块灰显/关内建留外接/恢复)。所有涉及拔屏的实验须用户知情同意、人在现场、先存盘。
- 系统显示 API 调用集中在 `CGDisplayService` 一处薄封装,便于隔离。

## 10. 已知风险与缓解

| 风险 | 缓解 |
|---|---|
| 私有符号跨 macOS 大版本可能改名/失效;非 Apple Silicon 平台未验证 | `isSupported = 符号经 dlsym 可用 且 hw.optional.arm64==1`;任一不满足 → 菜单顶部提示「不支持」+ 各屏只读不可切换(`toggle` 亦早退)。便于适配,也保证不在未验证平台动屏 |
| 崩溃/异常残留黑屏 | `.forAppOnly` 系统自动回滚 + 启动时恢复 + 退出时恢复 |
| 关到 0 块活跃屏(无恢复路径) | `canDisable` 机型感知:无内建屏机器须留 ≥1 活跃屏;有内建屏机器允许关到全黑(开盖恢复),但内建屏已被软件关掉/target 本身是内建屏时禁止 → 纯软件不会陷入无法恢复的全黑 |
| **死锁:软件关内建 + 物理拔光外接 → 持续全黑** | 纯软件路径已被 §7.1 堵死;仅余物理拔屏边界 → **强制重启即恢复(`.forAppOnly` 回滚 + 永不自启);软件自动恢复已证伪,不做** |
| 「有无内建屏」检测不准 | `hasBuiltInDisplay`:在线列表 + 内建电池代理;不确定时保守按「无内建屏」(更安全,宁可禁止全关) |
| 显示器热插拔导致菜单状态过期 | 菜单弹出时(`menuNeedsUpdate`)重新枚举重建 |
| `displayID` 随重连变化 | 持久标识用 UUID,不用 `displayID` |
